# 架构洞察：状态与控制流分离的设计哲学

> 这是一个针对 cclin Agent 架构演进的深入观察。我们在设计 `ApprovalManager` 与 `ToolOrchestrator` 时采用了与 `Session` 和 `ReactLoop` 完全同构的设计模式。本篇文档将解析这种底层相似性背后的软件工程哲学。

---

## 1. 核心洞察：策略与机制的隔离

在 cclin 的模块化演进中，我们始终遵循着一个经典的原则：**“策略与机制分离”（Separation of Mechanism and Policy）** 或者说 **“状态与控制流分离”**。

当系统面临复杂的业务流（如大模型的推断循环、工具的执行流水线）时，如果同时还要管理长效的“记忆”和“规则”（如上下文对话历史、工具调用权限许可），代码极其容易膨胀且难以测试。

我们的解法是：**把它们切分为“宏观大管家（状态容器）”与“微观执行者（控制引擎）”**。

## 2. 第一组映射：理解思考循环（Phase 2）

在 Agent 的思考底座中，我们将对话层拆分为了两个核心文件：

*   **`session.ts`（大管家 / 状态容器）**
    *   **职责**：持有对话历史（`_history`）、累计的 token 甚至当前的轮次状态。
    *   **特征**：它**很静**。它持有数据，知道我们“聊了什么”，但不知道 LLM 内部具体是如何在单次推断中打断、补写、调用 API 的。
    *   **生命周期**：跨越多轮对话，伴随程序的整个运行期。

*   **`react-loop.ts`（执剑人 / 控制引擎）**
    *   **职责**：包含复杂的 `while` 循环、JSON 解析、分支推断和多步工具分发逻辑。
    *   **特征**：它**很动**。它不长期保存数据，每一轮（Turn）只负责拿到传入的上下文在内部跑一个闭环，产出最终结果后立刻退出。
    *   **生命周期**：极短，仅存活于单轮问答（Turn）内部。

## 3. 第二组映射：理解工具执行流水线（Phase 4）

在把“执行工具”从直接调用变成了带有安全审批的流程后，我们复刻了上述的拆分模式：

*   **`ApprovalManager`（大管家 / 状态容器）**
    *   **职责**：持久化保存审批策略（always/once/session），以及记录过去的许可名单（基于指纹区分 `onceGrants` 和 `sessionGrants`）。
    *   **特征**：它也**很静**。它负责回答“这个操作的指纹以前被授权过吗？”，但完全不关心怎么打断程序、怎么在终端上展示字符。
    *   **生命周期**：跨越多个工具乃至多轮次，在高层维度上维持免密状态。

*   **`ToolOrchestrator`（执剑人 / 控制引擎）**
    *   **职责**：控制一个包含 6 个步骤（查找定位 → 触发验证与等待判定 → 解析入参 → 实际执行 → 错误拦截 → 结果截断）的复杂流水线。
    *   **特征**：它非常有**执行力**。它专注于怎么安全平稳地让一个动作落地。
    *   **生命周期**：极短，单次 `executeAction` 结束后就结束。

## 4. 这种同构设计的工程收益

### 收益一：控制流极速瘦身
含有 `while`、`try...catch`、`await` 和各种异常链路的流程控制代码，是最容易产生 Bug 的温床。将纯净的状态变更剥离出去后，流程引擎的代码能变得非常直觉化。Orchestrator 敢于在每一步抛出结果立刻截断，正是因为无需自己去推演状态的全局变化。

### 收益二：测试友好度极速攀升
*   测试 `ApprovalManager`：仅需 new 实例 -> 塞入 mock 数据并验证 Set 是否如期增加 -> 验证 true/false。无需 mock 整个底层的工具读写。
*   测试 `Session`：仅需验证 `runTurn` 前后历史数组的长短。

### 收益三：UI 彻底解耦（UI Agnosticism）
这也是这两个组合的绝妙之处：**它们共同把终端控制台（readline / console.log）阻挡在了核心逻辑层之外。**
*   `react-loop` 把 LLM 生成的文本结果“吐出”作为返回值，不自己做 `console.log`。
*   `ToolOrchestrator` 把“是否点击同意执行该工具”包装成 `Hooks.requestApproval` 丢给外部工厂处理，只等待返回的字符串。

不论将来是保留 CLI 终端，还是演进到 Ink TUI 的可视化弹窗，内核代码都无需做任何一行的修改。
