# 从“测评模型”转向“掌握 Agent 工具”：实战导向指南

## 0. 结论先行

你的想法是**对的，而且非常务实**：

- 在真实开发里，单纯比较“同模型在不同客户端谁更强”很难得出稳定结论。
- 变量太多（上下文注入、权限、工具集、会话压缩、项目结构、任务类型、时间压力、你的提示方式）。
- 因此，与其追求一个绝对排名，不如优先掌握 **Agent 工具使用能力**，把“可控变量”变成你的优势。

一句话：

> 模型上限决定天花板，工具与流程决定你每天能否稳定拿到结果。

---

## 1. 为什么“同模型测评”在实际场景很难做准

你前面提到的判断非常关键：**不同场景与项目会显著改变 LLM 表现**。具体原因可拆成 6 类。

### 1.1 任务分布不稳定

同一天里你可能做：

- 修 bug（定位 + 回归）
- 新功能（设计 + 改造）
- 重构（行为不变）
- 文档与测试补齐

不同任务对模型要求完全不同，单次结果很难横向比较。

### 1.2 上下文质量差异巨大

同模型在两端表现差异，常常不是“智力差”，而是“喂给它的上下文差”。

- IDE 自动注入：当前文件、选区、错误面板、项目结构
- CLI 手工注入：你是否精确传入文件、日志、失败用例

上下文一旦偏差，回答质量会阶跃式下降。

### 1.3 工具权限与执行闭环不同

会不会读写文件、会不会跑测试、能否拿到诊断、是否要人工批准，都会直接影响“执行质量”。

### 1.4 系统指令与产品策略不同

即使底层模型名相同，不同产品壳层也会有：

- 不同 system/developer 指令
- 不同安全策略
- 不同输出格式约束

最终表现自然不同。

### 1.5 会话管理差异

长对话会被压缩/截断，不同客户端压缩策略不同，导致“中后期智商波动”。

### 1.6 人机协作习惯是最大隐变量

同一模型，同一任务，不同人引导方式差别巨大。你会不会：

- 给出验收标准
- 强制先读关键文件
- 要求先列计划再改代码
- 要求改完自测并汇报

这些往往比模型切换更影响结果。

## 2. 实战策略：从“模型崇拜”转为“工具编排”

### 2.1 核心目标

将“偶尔惊艳”变成“稳定交付”：

- 结果可复现
- 过程可审计
- 失败可回滚
- 效率可持续

### 2.2 你真正要优化的对象

不是“哪个模型回答更像人”，而是：

$$
	ext{交付质量} = f(\text{任务拆解},\ \text{上下文注入},\ \text{工具链},\ \text{验证闭环},\ \text{风险控制})
$$

---

## 3. Agent 工具能力地图（按实战价值排序）

下面是通用能力，不绑定某个厂商。

### 3.1 信息获取类（输入质量）

1. 项目级搜索（语义/全文）
2. 文件结构与依赖关系定位
3. 错误与测试结果采集
4. 运行时日志采集

目标：让 Agent“看到正确问题”。

### 3.2 变更执行类（输出质量）

1. 精确编辑（最小改动）
2. 批量改动（多文件一致性）
3. 代码生成与重构
4. 配置/脚本变更

目标：让 Agent“改对且改小”。

### 3.3 验证闭环类（可靠性）

1. 单测/集成测试执行
2. Lint/类型检查
3. 回归验证
4. 失败重试与定位

目标：让 Agent“自证正确”。

### 3.4 协作治理类（可控性）

1. 计划与里程碑跟踪
2. 审批模式（手动/自动）
3. 会话历史与回滚
4. 提交说明与 PR 描述

目标：让 Agent“可管理、可追责”。

## 4. 一套可落地的 Agent 工作流（建议默认）

把每个任务固定成 6 步。

### 步骤 1：定义任务合同（1 分钟）

写清 4 件事：

- 目标（要解决什么）
- 范围（改哪些，不改哪些）
- 验收（哪些测试必须过）
- 约束（性能/风格/兼容策略）

### 步骤 2：先读后改

要求 Agent 先：

- 列出将读取的关键文件
- 说明推断路径
- 不允许“拍脑袋直接改”

### 步骤 3：先计划后执行

计划至少包含：

- 子任务清单
- 风险点
- 验证方式

### 步骤 4：最小变更实施

规则：

- 每次改动围绕单一目的
- 避免无关重排与大面积格式化
- 保持 API 稳定，除非需求明确要求破坏式修改

### 步骤 5：强制验证

至少跑：

- 受影响测试
- 静态检查
- 关键路径冒烟

### 步骤 6：结构化汇报

固定输出：

1. 改了什么
2. 为什么这么改
3. 怎么验证的
4. 还有哪些风险

## 5. 提示词模板：把“会聊天”变成“会交付”

以下模板可直接套用（按你的风格再微调）。

### 5.1 实施任务模板

请按以下流程执行：

1) 先读取并列出你将参考的文件；
2) 给出 5-8 条可执行计划，不要改代码；
3) 我确认后再进行最小改动实现；
4) 改完后运行测试/检查，并给出失败与修复过程；
5) 最后用“变更摘要 + 验证结果 + 风险点”格式汇报。

约束：禁止无关重构，禁止修改未涉及模块。

### 5.2 排障模板

目标：定位并修复 [现象]。

先做：

- 假设列表（按概率排序）
- 每个假设对应的验证动作

然后：

- 执行最小验证，逐步排除
- 在确认根因前不做大改
- 修复后给出复现步骤与防回归建议

### 5.3 重构模板

目标：重构 [模块名]，保持外部行为不变。

要求：

- 先定义不变量与回归点
- 每轮只做一种重构动作
- 每轮结束必须可编译、可测试
- 输出前后复杂度/可读性变化说明

## 6. 如何训练“工具掌握度”：4 周进阶法

### 第 1 周：检索与定位

目标：让 Agent 快速找对上下文。

- 每天 3 个“定位题”（函数定义、调用链、配置入口）
- 要求输出定位依据，不接受只给结论

### 第 2 周：最小改动

目标：提高改动精准度。

- 每天 2 个小 bug 修复
- 限制单次改动文件数与行数
- 关注“改动最小但测试通过”

### 第 3 周：验证闭环

目标：培养“改完即证”的习惯。

- 每次必须附验证命令与结果摘要
- 失败时要求给出下一步排查分支

### 第 4 周：协作与治理

目标：可审计、可回滚。

- 统一报告模板
- 统一计划模板
- 引入审批策略（高风险改动人工确认）

---

## 7. 你真正该追踪的指标（比“主观感觉”更有用）

建议用这 8 个指标观察两周：

1. 一次通过率（不返工）
2. 平均修复轮数
3. 首次可运行时间
4. 测试通过率
5. 回归率（改 A 坏 B）
6. 平均改动规模（文件数/行数）
7. 人工介入次数
8. 任务总时长

如果这些指标改善，即使你没换模型，也是在变强。

---

## 8. 常见误区与纠偏

### 误区 1：频繁换模型

纠偏：先把任务合同、上下文注入、验证闭环标准化，再谈换模型。

### 误区 2：一次给超大目标

纠偏：拆成可验证的小步；每步都要有“完成定义”。

### 误区 3：只看答案，不看过程

纠偏：强制要求 Agent 报告“依据、改动、验证、风险”。

### 误区 4：把 Agent 当“自动驾驶”

纠偏：高风险操作保留人工批准；把 Agent 当“副驾驶 + 自动化执行器”。

---

## 9. 何时再做“模型对比测评”才有意义

当你已经做到：

- 固定流程（任务合同 + 计划 + 最小改动 + 验证）
- 固定数据（同仓库、同任务、同约束）
- 固定口径（统一评分表）

这时再做 A/B 才有参考价值。否则结论大概率是噪声。

---

## 10. 最终建议（针对你当前思路）

你的方向可以总结为：

1. 接受“真实世界很难做绝对模型排名”这件事；
2. 把精力转向“可控变量”——工具、流程、验证；
3. 先建立你的 Agent 标准作业法，再谈模型优劣。

这是从“好奇哪家更聪明”走向“稳定产出结果”的关键一步。
