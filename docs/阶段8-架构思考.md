>个人思考：
- 问题
Session 初始化需要 tuiMiddleware，但 middleware 只有在 React 组件渲染完成后才能产生；App 渲染提交时又需要 session.runTurn，两边互相等待，形成死锁。

- 解法
声明了一个顶层的 session 做间接层，允许为值 null。把 App 对 session.runTurn 的硬依赖（props直接传值，导致直接捕获当前为实例化的 session），改成传一个闭包函数——闭包捕获的是变量的引用而不是当前值，所以传出去时 session 是 null 也没问题。让 session.runTurn 从构造时，变为在运行时。

- 为什么安全
App 渲染完成后 useEffect 立即触发，调用回调创建 Session 并赋值给那个变量。用户的键盘输入在物理上一定晚于此，所以 session.runTurn 被调用时变量必然已有值。

# Phase 8 架构学习笔记：TUI 升级与桥接模式

## 背景与痛点

在 Phase 8 的重构中，目标是将简单的 `readline` CLI 升级为使用 Ink 驱动的交互式 Terminal User Interface (TUI)。这引入了一个经典的循环依赖问题：“鸡生蛋问题”。

1. **Session 的需求**：`Session` 需要外部传入 `middlewares` 参数才能进行构造（Hook 注册表在构造时就已构建）。
2. **App 组件的需求**：Ink 的 `App` 组件内部通过 React 的 `useMemo` 创建了 `tuiMiddleware`，用于连接 Hook 和 UI 状态更新。并且 `App` 组件提供了 `onSubmit` 回调（当用户按下 Enter 键时），该回调内部需要调用 `session.runTurn(input)`。
3. **矛盾点**：创建 `Session` 需要 `App` 渲染后提供的 `tuiMiddleware`；而 `App` 的提交操作需要已创建好的 `Session`。

## 解决方案：桥接模式（Bridge Pattern）

针对这个循环依赖，当前的架构采用了一种**延迟绑定+闭包引用（桥接模式）**的解法，将初始化链路分为两拍：

### 第一阶段：初始化外部资源与渲染 TUI

在入口文件 `src/index.ts` 中：
1. **准备基础设施**：首先创建好不受循环依赖影响的独立组件，例如 `ToolRouter`（内置和 MCP 工具）、`CallLLM`、系统 Prompt、`ApprovalManager` 以及 `TokenCounter`。
2. **声明桥接变量**：在 `index.ts` 顶层作用域声明 `let session: Session | null = null`，以及一些闭包引用的回调（如 `requestApprovalFn`、`onAssistantChunkFn`）。此时 `session` 还未实例化。
3. **渲染 App 组件**：调用 `render(<App ... />)` 挂载 Ink 界面。注意，在 `App` 标签的参数中，我们传入了 `onSubmit` 回调和一些 `onReady` 系列的注册函数。

> **为什么必须传闭包，而不能直接传 `session.runTurn` 作为 prop？**
>
> 因为在调用 `render(<App ... />)` 的那一刻，`session` 仍然是 `null`。如果写成 `onSubmit={session.runTurn}`，JavaScript 会立即对 `session.runTurn` 求值——此时 `session` 是 `null`，读取 `null.runTurn` 会直接抛出 TypeError，代码根本无法走到渲染步骤。
>
> 而传入 `handleSubmit` 闭包则完全不同：闭包只是对外部 `session` **变量的引用**，传递的瞬间不会读取其值。只有用户真正提交消息、触发 `handleSubmit` 时，函数体才会执行并读取 `session`——彼时 `useEffect` 已经回调过、`Session` 早已实例化，安全无误。

```typescript
// src/index.ts 桥接声明示例
let session: Session | null = null

const handleSubmit = async (input: string) => {
    // 闭包引用外部可变的 session 变量
    if (!session) return
    await session.runTurn(input)
}

const handleMiddlewareReady = (mw: AgentMiddleware) => {
    // Session 在中间件就绪后才被创建
    session = new Session({
        /* ..., middlewares: [mw] */
    })
}
```

### 第二阶段：App 挂载完成后的反馈

在 `src/tui/app.tsx` 组件内部：
1. 使用 `useMemo` 实例化依赖 React `useReducer` action `dispatch` 函数的 `tuiMiddleware`。
2. 利用 `useEffect` 钩子会在组件首次完成渲染后同步触发的特性，在副作用中调用 `onMiddlewareReady(tuiMiddleware)` 等一系列 `onReady` 回调。

```tsx
// src/tui/app.tsx
const tuiMiddleware = React.useMemo<AgentMiddleware>(() => ({
    name: 'tui',
    onTurnStart: ({ turn, input }) => { /* dispatch state ... */ },
    // ...
}), [dispatch])

React.useEffect(() => {
    onMiddlewareReady(tuiMiddleware)
    // ... 其他 ready 钩子
}, [tuiMiddleware, onMiddlewareReady])
```

### 为什么这个方案能工作？

React 的 `useEffect` 会在首次挂载后理解触发回调，**此时用户根本还来不及进行任何键盘输入并触发 `onSubmit` 事件**。因此：
1. `App` 先挂载，并在 `useEffect` 中马上调用把 `tuiMiddleware` 传递给外侧 `index.ts` 的回调函数中。
2. 此时 `handleMiddlewareReady` 立即执行，`new Session(...)` 实例化并把应用级别的实例赋给闭包里的 `session` 变量。
3. 等用户终于敲击键盘发出一次 `input` 从而触发 `handleSubmit` 时，`session` 变量早已经被实例化，`!session` 前置防御被短路，安全地调用 `session.runTurn(input)`。

通过 `let session` 闭包与明确界定的控制流顺序有效解除了循环依赖。

## 阶段 8 的组件化拆分结构

基于上述桥接模式，TUI 的文件设计体现出了明确的分工：

### `index.ts` (初始化层与控制反转中心)
* **职责**：整个系统的根入口，负责统筹配置的加载（`.env` 等）、服务的启动组装，充当各子组件之间的连接“胶水”层。
* **特性**：它本身不负责任何 UI 的绘制和 LLM 具体执行，但它维护所有的“桥梁”（例如上面讲解的闭包引用），将底层的 `Session`/`ToolOrchestrator` 与顶层的 React `App` 无缝组装起来。

### `tui/app.tsx` (应用级状态与中间件工厂)
* **职责**：Ink 界面的根组件，扮演着 Controller 角色和 TUI Root。
* **特性**：
    * 利用 `useReducer`（搭配 `chatTimelineReducer`）维护全局所有的对话 UI 状态表（历史 Turn、进行中的步骤、系统消息、繁忙状态等）。
    * 创建连接 Runtime 事件和 UI 的 `tuiMiddleware` 并向外“汇报”。
    * 将具体状态当作 Props 下发给展示类子组件（如 `OutputArea` 和 `InputArea`）。

### `tui/output.tsx` (输出时间线渲染层)
* **职责**：无状态（更偏负责展示）的纯视图组件，渲染应用中的内容层。
* **特性**：
    * 将状态数据映射为 Ink 界面。包含不同粒度的细胞组件 (`SystemCell`, `StepCell`, `TurnCell`)。
    * 对话记录渲染。采用 Ink 的 `<Static>` 组件渲染已冻结（确定完成）的历史对话来提高终端渲染效率。当前进行中的回合（活动回合）则在通过 React 动态刷新以实现流式输出效果。

### `tui/input.tsx` (输入控制与交互终端)
* **职责**：管理用户键盘输入交互（REPL 提示符区域及其状态）。
* **特性**：
    * 自定义了终端键盘操作光标交互操作（左右移动、Backspace，快捷键）的编辑器缓冲区（使用 useState 管理 value 与 cursor 位置）。
    * 处理带有 UI 阻塞性质的情况，如工作进行时的显示、审批 overlay 界面的覆盖与应答处理（Y / N）。
    * 提供 Slash command `/` 建议和补全功能。

## 为什么这是「架构问题」，而不只是「时序问题」

表面上看，解决方案修复的是"谁先创建"的时序冲突，但循环依赖的根源是两个模块的**职责边界没有划清**：

- `Session`（runtime 层）想拿到 `tuiMiddleware` → 隐式依赖了 UI 层的产物
- `App`（UI 层）想直接调用 `session.runTurn` → 隐式依赖了 runtime 层的实例

两层互相侵入，谁也不能独立存在，这是**分层架构被破坏**的典型表现。

解法实际做了两件事，后者才是真正的「架构调整」：

| 手段 | 解决的问题 |
|---|---|
| 闭包 + 延迟实例化 | **时序问题**：打破初始化死锁，让两边都能在"对方还没好"时安全启动 |
| 将所有依赖收拢到 `index.ts` | **架构问题**：Session 不认识 App，App 不认识 Session，两者只认识 `index.ts` 传给各自的回调函数 |

```
之前（概念上）：
  Session ←→ App   （双向耦合）

之后：
  Session ← index.ts → App   （各自只与 index.ts 交互，解耦）
```

`index.ts` 因此承担起**控制反转（IoC）中心**的角色——它不画 UI、不执行 LLM，只负责把各层组装起来。这个职责本身就是主动的架构决策，不是「顺带调整了代码」。

---

## 总结

Phase 8 利用 **桥接与延迟实例化模式** 和原生的 **React Hook 生命周期配合（`useEffect` 同步触发）**，优雅地连接了要求依赖中间件的重业务状态管理方 (`Session`) 与只能在渲染周期内出产 UI Store Callback 中间件视图呈现方 (`App`)。

更深层的意义在于：`index.ts` 被设计为**控制反转（IoC）的中间层**——Session 与 App 都只与它交互，彼此完全解耦。闭包是打破时序死锁的手段，而分层解耦才是这次架构调整的核心目标。

这种自顶向下拆分的架构能够良好的应对日益增长的业务需求（例如之后增加支持基于 Terminal 的 Markdown 排版展示、异步并行展示系统等）。
