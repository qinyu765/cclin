# CLI 工具中纯 JS 与 WASM 的选择：以 Tokenizer 为例

> 摘要：本文整理了为在开发 CLI 工具时，为什么往往选择纯 JS 的 tokenizer 库（如 `gpt-tokenizer`）而不是性能更好的 WASM 原生绑定库（如 `@dqbd/tiktoken`）的核心考量因素。

在开发像 `cclin` 这样的 CLI（命令行）工具时，选择 **`gpt-tokenizer`（纯 JS）** 而放弃速度理论上更快的 **`@dqbd/tiktoken`（WASM 绑定）**，最核心的原因是**最大化分发与安装的成功率（即“零摩擦”的开发者体验）**。

具体原因可以拆解为以下几点：

---

### 1. 告别跨平台安装噩梦（最重要的原因）

WASM 原生绑定或 C++ 插件（如用到 `node-gyp` 的情况）经常是 Node.js 开发者安装全局工具时的噩梦：

* 很多用户的电脑里并没有装完整的 C++ 编译环境（如 Visual Studio Build Tools 或 Xcode Command Line Tools）。
* 即使提供预编译好的二进制包（Prebuilt binaries），也容易在奇怪的 CPU 架构（比如特殊的 ARM 设备环境）或较新的 Node.js 版本上找不到对应版本而安装失败。
* 纯 JS 库在执行 `npm install -g cclin` 时，**只要有 Node 就能装上，不管你是 Windows、Mac、还是树莓派**，100% 成功，即插即用，零编译。

---

### 2. 性能差异在当前场景下可忽略不计

对于统计 Token 和做简单的历史截断（比如几千到 10 万 Token 的级别）：

* `@dqbd/tiktoken` 的 WASM 版本可能只需要 **1ms**。
* `gpt-tokenizer` 的纯 JS 版本可能需要 **10-15ms**。
* 但是，核心耗时（瓶颈）是调用 LLM 的网络请求（通常是 **数秒 甚至 十几秒**），这十几毫秒在网络延迟面前连零头都算不上。为了连感知都没有的性能优化去承担安装失败的风险，非常不划算。

---

### 3. CLI 工具的冷启动优势

加载和初始化一个巨大的 WASM 模块，有时候在冷启动开销上反而会吃点亏（特别是在频繁按下回车、执行短命令的 CLI 中）。纯 JS 的按需加载和现代 V8 引擎的 JIT（即时编译）优化已经足够快，启动和运行非常轻量。

---

### 4. 环境兼容性强（如边缘计算/运行库切换）

纯 JS 的库不依赖任何底层系统 API，未来如果你想把这段核心逻辑抽离，运行在浏览器、Cloudflare Workers (Edge Computing) 或者切换运行时到 Deno、Bun 中，都不需要面临 WASM 环境支持的问题，具备无缝迁移的能力。

---

**总结**

对于一个主打轻量、好用的脚手架或 CLI 来说，“让每一位用户按回车就能无痛安装成功”的优先级，远高出“解析字符串快出几毫秒”的优先级。
